# AI 需求理解与业务范式分析

## 一、需求理解

### 核心问题

企业内部多个业务系统在关键事件发生时，需要调用**不同外部供应商的 HTTP(S) API** 进行通知。这些外部 API 的地址、Header/Body 格式各不相同，但业务系统本身**不关心返回值，只关心通知能否可靠送达**。

### 需要解决的核心痛点

1. **解耦** — 业务系统不应直接对接各种外部 API，否则每新增一个供应商就要改业务代码
2. **可靠投递** — 外部系统可能失败或不可用，业务系统不想为此操心，需要一个中间层来保证"通知一定能送到"

### 系统定位

本质上是一个 **HTTP 回调/通知网关服务**（类似 Webhook Delivery 服务），职责是：

| 属于本系统的职责 | 不属于本系统的职责 |
|---|---|
| 接收业务系统的通知请求并持久化 | 解析/处理外部 API 的返回值 |
| 按不同供应商的格式要求组装请求 | 业务事件本身的触发逻辑 |
| 失败重试（含退避策略） | 供应商 API 的业务语义理解 |
| 投递状态追踪与告警 | 供应商认证/密钥的安全管理（可协商） |
| 长期不可用时的死信/降级处理 | |

### 关键设计决策需要回答的问题

1. **系统边界** — 什么做、什么不做，以及不做的理由
2. **投递语义** — "至少一次"（at-least-once）是最务实的选择，配合幂等机制由消费方保障
3. **失败处理** — 重试策略（指数退避）、最大重试次数、死信队列、告警机制

### 整体判断

这是一个典型的 **异步可靠消息投递** 场景，核心挑战在于：

- **可靠性**：通过持久化 + 重试保证消息不丢
- **适配性**：不同供应商的请求格式需要可配置/可插拔
- **可观测性**：投递状态必须可查，异常必须可感知

## 二、业内实现范式

### 1. 经典架构范式：Outbox Pattern（发件箱模式）

这是解决"业务操作 + 通知投递"一致性问题最核心的模式：

```
业务系统 → 写业务数据 + 写 Outbox 表（同事务）→ 通知服务轮询/监听 Outbox → 投递外部 API
```

**为什么重要**：如果先写业务数据再发消息，消息可能丢；如果先发消息再写业务数据，业务可能失败。Outbox 把两者绑定在同一个数据库事务里，保证"业务发生了，通知就一定在"。

### 2. 投递引擎的典型实现流派

#### 流派 1：定时轮询型

```
Outbox 表 → 定时扫描未投递记录 → HTTP 投递 → 更新状态
```

- 代表：中小规模内部系统
- 优点：实现简单，依赖少
- 缺点：轮询有延迟，大量记录时性能差

#### 流派 2：消息队列解耦型

```
Outbox 表 → CDC（Debezium/Canal）→ Kafka/RabbitMQ → 消费者投递
```

- 代表：Stripe Webhooks、中型企业
- 优点：天然背压、水平扩展、解耦彻底
- 缺点：引入 MQ 和 CDC，运维成本上升

#### 流派 3：专用 Webhook 服务型

```
业务系统 → 调用 Webhook 服务 API → 服务内部持久化 + 投递
```

- 代表：GitHub Webhooks、Twilio、Svix（开源 Webhook 即服务）
- 优点：投递逻辑集中管控，业务系统零耦合
- 缺点：需要自建或引入第三方

**本项目场景最适合流派 3**，因为需求描述的就是"接收业务系统请求，代为投递"。

### 3. 可靠性保障的行业标准做法

#### 3.1 投递语义：At-Least-Once（至少一次）

- 业内几乎全部选择 at-least-once，而非 exactly-once
- 原因：HTTP 调用天生不具备两阶段提交能力，exactly-once 在跨网络场景下不可行
- 副作用：消费方可能收到重复通知 → **要求消费方实现幂等**

#### 3.2 重试策略：Exponential Backoff + Jitter

```
1st retry:  ~10s
2nd retry:  ~30s
3rd retry:  ~1min
4th retry:  ~5min
5th retry:  ~15min
6th retry:  ~1h
...
```

- **Jitter（随机抖动）**是关键：防止所有失败请求在同一时刻重试，产生"惊群效应"
- Stripe、AWS EventBridge 都采用此策略

#### 3.3 Circuit Breaker（熔断器）

- 当某个供应商连续失败 N 次，临时"熔断"，不再盲目重试
- 熔断期间走半开探测，恢复后重新放行
- 防止外部系统故障时本系统被拖垮（线程池耗尽、连接池打满）

#### 3.4 Dead Letter Queue（死信队列）

- 超过最大重试次数仍未成功的通知 → 进入死信
- 死信不丢弃，而是**持久化保存 + 告警**
- 运维人员可手动排查、修复后重新投递

#### 3.5 投递状态机

```
PENDING → SENDING → SUCCESS
                   → FAILED → RETRYING → ... → DEAD_LETTER
```

- 每次状态变更都记录时间戳、重试次数、最后错误信息
- 可查询、可追踪、可告警

### 4. 供应商适配的业内做法

#### 模板化配置

```yaml
vendors:
  ad-system:
    url: https://ad.example.com/callback
    method: POST
    headers:
      Authorization: Bearer ${api_key}
      Content-Type: application/json
    body_template: |
      {"event": "${event_type}", "user_id": "${user_id}", "ts": "${timestamp}"}
```

- 用模板引擎渲染最终请求
- 新增供应商**不改代码，只加配置**

#### 签名机制

- 对请求体做 HMAC 签名，放在 Header 中
- 让消费方能验证请求确实来自本系统，且未被篡改
- Stripe Webhooks 的 `stripe-signature` Header 就是这个模式

### 5. 可观测性

| 维度 | 做法 |
|---|---|
| 投递状态 | 每条通知可查询当前状态 + 历史投递记录 |
| 指标监控 | 投递成功率、平均延迟、重试次数分布 |
| 告警 | 死信产生、熔断触发、队列积压 |
| 链路追踪 | 通知 ID 贯穿全链路，可从业务 ID 查到所有关联通知 |

### 6. 开源参考实现

| 项目 | 特点 |
|---|---|
| **Svix** | 开源 Webhook 即服务，Go 实现，功能最完整 |
| **Supabase Edge Functions** | 内置 Webhook 投递，Postgres 驱动 |
| **Outboxer** | Go 语言的通用 Outbox 模式库 |

### 7. 针对本项目的推荐范式

结合需求特征（内部服务、多供应商适配、可靠投递、业务不关心返回值），最佳组合是：

1. **独立通知服务**（流派 3），业务系统只需调一个统一 API
2. **持久化优先**：先落库，再投递，保证消息不丢
3. **At-Least-Once + 幂等建议文档**：在 API 文档中明确告知消费方需幂等
4. **指数退避 + 熔断 + 死信**：三件套保障极端情况下的系统稳定性
5. **模板化供应商配置**：新增供应商零代码变更

## 三、系统边界说明

> 本系统的边界是**"可靠地将通知投递到指定供应商 API"**，边界之外的问题——业务语义、安全存储、流量管控、结果回调——要么属于其他系统的基础设施职责，要么会模糊本系统的定位，要么在当前规模下收益不足以覆盖复杂度。

### 1. 选择在本系统中解决的问题

| 问题 | 解决方式 |
|---|---|
| **多供应商 API 适配** | 模板化配置（vendor_configs），新增供应商零代码变更，只加配置 |
| **通知可靠投递** | 先落库再投递（Outbox Pattern）+ At-Least-Once 语义，保证消息不丢 |
| **投递失败重试** | 指数退避 + Jitter 重试策略，避免惊群效应，最多 6 次重试 |
| **外部系统故障保护** | 每供应商独立熔断器，防止故障供应商拖垮本系统 |
| **死信兜底** | 超过最大重试或 4xx 客户端错误进入死信，不丢弃，支持手动重投递 |
| **投递可追踪** | 每次投递尝试记录 delivery_logs，通知状态全程可查 |
| **投递延迟** | PG LISTEN/NOTIFY 实时触发（~500ms）+ 轮询兜底（5s），双重保障 |
| **多实例并发安全** | PG 行级 CAS 抢占，同一条通知只被一个实例投递 |
| **供应商运行时管理** | 供应商配置 CRUD API，增删改无需重启服务 |

### 2. 明确选择不解决的问题及原因

| 不解决的问题 | 原因 |
|---|---|
| **外部 API 返回值的业务语义处理** | 需求明确说"业务系统不需要关心外部 API 的返回值"。本系统只关心投递是否成功（2xx），不解析响应体中的业务含义。如果未来需要，应由业务系统自己订阅投递结果回调，而非本系统代为处理 |
| **Exactly-Once 投递语义** | HTTP 调用天生不具备两阶段提交能力，exactly-once 在跨网络场景下不可实现。At-Least-Once + 消费方幂等是业内唯一务实的选择 |
| **供应商认证密钥的安全存储** | 供应商密钥（`vendor_configs.secret`）当前以明文存库。加密存储、密钥轮换、Vault 集成等属于基础设施安全层面的能力，不应耦合在通知服务中，应由企业的密钥管理基础设施统一解决 |
| **业务事件的触发逻辑** | 本系统是被动接收者（业务系统调 API 提交通知），不负责"何时该发通知"的判断。比如"用户注册后该通知哪个广告系统"属于业务规则，应由业务系统决定 |
| **请求限流 / 流量控制** | 当前未实现按业务系统维度的限流。限流策略与业务强相关（不同业务方配额不同），适合在 API 网关层统一管控，而非通知服务内部实现 |
| **消息优先级** | 所有通知按 `next_retry_at` 时间顺序投递，不支持业务方指定优先级。优先级调度会显著增加系统复杂度（优先级队列饥饿问题），当前业务场景无此强需求 |
| **投递结果的业务回调** | 本系统不主动回调业务系统告知投递结果。业务系统可通过 `GET /api/v1/notifications/{id}` 主动查询。主动回调意味着需要管理回调地址、重试、幂等，本质上是另一个通知投递问题，会让系统职责模糊 |
| **消息内容的校验与过滤** | 本系统不对 `payload` 内容做业务合法性校验，只负责透传到模板渲染。内容合规审查（如敏感词过滤）属于业务域职责 |
| **集群级别的分布式锁** | 当前通过 PG 行级 CAS 保证同一通知只被一个实例投递，未引入 Redis 分布式锁。PG CAS 在当前规模下足够，引入 Redis 会增加运维复杂度和故障点。如果未来实例数超过 10+，再考虑引入 |
| **投递顺序保证** | 同一业务方的多条通知不保证严格有序。保证有序需要串行化投递，会严重影响吞吐量。消费方应通过 `biz_id` + 时间戳自行处理乱序 |
