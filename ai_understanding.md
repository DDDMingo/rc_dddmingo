# AI 需求理解与业务范式分析

## 一、需求理解

### 核心问题

企业内部多个业务系统在关键事件发生时，需要调用**不同外部供应商的 HTTP(S) API** 进行通知。这些外部 API 的地址、Header/Body 格式各不相同，但业务系统本身**不关心返回值，只关心通知能否可靠送达**。

### 需要解决的核心痛点

1. **解耦** — 业务系统不应直接对接各种外部 API，否则每新增一个供应商就要改业务代码
2. **可靠投递** — 外部系统可能失败或不可用，业务系统不想为此操心，需要一个中间层来保证"通知一定能送到"

### 系统定位

本质上是一个 **HTTP 回调/通知网关服务**（类似 Webhook Delivery 服务），职责是：

| 属于本系统的职责 | 不属于本系统的职责 |
|---|---|
| 接收业务系统的通知请求并持久化 | 解析/处理外部 API 的返回值 |
| 按不同供应商的格式要求组装请求 | 业务事件本身的触发逻辑 |
| 失败重试（含退避策略） | 供应商 API 的业务语义理解 |
| 投递状态追踪与告警 | 供应商认证/密钥的安全管理（可协商） |
| 长期不可用时的死信/降级处理 | |

### 关键设计决策需要回答的问题

1. **系统边界** — 什么做、什么不做，以及不做的理由
2. **投递语义** — "至少一次"（at-least-once）是最务实的选择，配合幂等机制由消费方保障
3. **失败处理** — 重试策略（指数退避）、最大重试次数、死信队列、告警机制

### 整体判断

这是一个典型的 **异步可靠消息投递** 场景，核心挑战在于：

- **可靠性**：通过持久化 + 重试保证消息不丢
- **适配性**：不同供应商的请求格式需要可配置/可插拔
- **可观测性**：投递状态必须可查，异常必须可感知

## 二、业内实现范式

### 1. 经典架构范式：Outbox Pattern（发件箱模式）

这是解决"业务操作 + 通知投递"一致性问题最核心的模式：

```
业务系统 → 写业务数据 + 写 Outbox 表（同事务）→ 通知服务轮询/监听 Outbox → 投递外部 API
```

**为什么重要**：如果先写业务数据再发消息，消息可能丢；如果先发消息再写业务数据，业务可能失败。Outbox 把两者绑定在同一个数据库事务里，保证"业务发生了，通知就一定在"。

### 2. 投递引擎的典型实现流派

#### 流派 1：定时轮询型

```
Outbox 表 → 定时扫描未投递记录 → HTTP 投递 → 更新状态
```

- 代表：中小规模内部系统
- 优点：实现简单，依赖少
- 缺点：轮询有延迟，大量记录时性能差

#### 流派 2：消息队列解耦型

```
Outbox 表 → CDC（Debezium/Canal）→ Kafka/RabbitMQ → 消费者投递
```

- 代表：Stripe Webhooks、中型企业
- 优点：天然背压、水平扩展、解耦彻底
- 缺点：引入 MQ 和 CDC，运维成本上升

#### 流派 3：专用 Webhook 服务型

```
业务系统 → 调用 Webhook 服务 API → 服务内部持久化 + 投递
```

- 代表：GitHub Webhooks、Twilio、Svix（开源 Webhook 即服务）
- 优点：投递逻辑集中管控，业务系统零耦合
- 缺点：需要自建或引入第三方

**本项目场景最适合流派 3**，因为需求描述的就是"接收业务系统请求，代为投递"。

### 3. 可靠性保障的行业标准做法

#### 3.1 投递语义：At-Least-Once（至少一次）

- 业内几乎全部选择 at-least-once，而非 exactly-once
- 原因：HTTP 调用天生不具备两阶段提交能力，exactly-once 在跨网络场景下不可行
- 副作用：消费方可能收到重复通知 → **要求消费方实现幂等**

#### 3.2 重试策略：Exponential Backoff + Jitter

```
1st retry:  ~10s
2nd retry:  ~30s
3rd retry:  ~1min
4th retry:  ~5min
5th retry:  ~15min
6th retry:  ~1h
...
```

- **Jitter（随机抖动）**是关键：防止所有失败请求在同一时刻重试，产生"惊群效应"
- Stripe、AWS EventBridge 都采用此策略

#### 3.3 Circuit Breaker（熔断器）

- 当某个供应商连续失败 N 次，临时"熔断"，不再盲目重试
- 熔断期间走半开探测，恢复后重新放行
- 防止外部系统故障时本系统被拖垮（线程池耗尽、连接池打满）

#### 3.4 Dead Letter Queue（死信队列）

- 超过最大重试次数仍未成功的通知 → 进入死信
- 死信不丢弃，而是**持久化保存 + 告警**
- 运维人员可手动排查、修复后重新投递

#### 3.5 投递状态机

```
PENDING → SENDING → SUCCESS
                   → FAILED → RETRYING → ... → DEAD_LETTER
```

- 每次状态变更都记录时间戳、重试次数、最后错误信息
- 可查询、可追踪、可告警

### 4. 供应商适配的业内做法

#### 模板化配置

```yaml
vendors:
  ad-system:
    url: https://ad.example.com/callback
    method: POST
    headers:
      Authorization: Bearer ${api_key}
      Content-Type: application/json
    body_template: |
      {"event": "${event_type}", "user_id": "${user_id}", "ts": "${timestamp}"}
```

- 用模板引擎渲染最终请求
- 新增供应商**不改代码，只加配置**

#### 签名机制

- 对请求体做 HMAC 签名，放在 Header 中
- 让消费方能验证请求确实来自本系统，且未被篡改
- Stripe Webhooks 的 `stripe-signature` Header 就是这个模式

### 5. 可观测性

| 维度 | 做法 |
|---|---|
| 投递状态 | 每条通知可查询当前状态 + 历史投递记录 |
| 指标监控 | 投递成功率、平均延迟、重试次数分布 |
| 告警 | 死信产生、熔断触发、队列积压 |
| 链路追踪 | 通知 ID 贯穿全链路，可从业务 ID 查到所有关联通知 |

### 6. 开源参考实现

| 项目 | 特点 |
|---|---|
| **Svix** | 开源 Webhook 即服务，Go 实现，功能最完整 |
| **Supabase Edge Functions** | 内置 Webhook 投递，Postgres 驱动 |
| **Outboxer** | Go 语言的通用 Outbox 模式库 |

### 7. 针对本项目的推荐范式

结合需求特征（内部服务、多供应商适配、可靠投递、业务不关心返回值），最佳组合是：

1. **独立通知服务**（流派 3），业务系统只需调一个统一 API
2. **持久化优先**：先落库，再投递，保证消息不丢
3. **At-Least-Once + 幂等建议文档**：在 API 文档中明确告知消费方需幂等
4. **指数退避 + 熔断 + 死信**：三件套保障极端情况下的系统稳定性
5. **模板化供应商配置**：新增供应商零代码变更

## 三、系统边界说明

> 本系统的边界是**"可靠地将通知投递到指定供应商 API"**，边界之外的问题——业务语义、安全存储、流量管控、结果回调——要么属于其他系统的基础设施职责，要么会模糊本系统的定位，要么在当前规模下收益不足以覆盖复杂度。

### 1. 选择在本系统中解决的问题

| 问题 | 解决方式 |
|---|---|
| **多供应商 API 适配** | 模板化配置（vendor_configs），新增供应商零代码变更，只加配置 |
| **通知可靠投递** | 先落库再投递（Outbox Pattern）+ At-Least-Once 语义，保证消息不丢 |
| **投递失败重试** | 指数退避 + Jitter 重试策略，避免惊群效应，最多 6 次重试 |
| **外部系统故障保护** | 每供应商独立熔断器，防止故障供应商拖垮本系统 |
| **死信兜底** | 超过最大重试或 4xx 客户端错误进入死信，不丢弃，支持手动重投递 |
| **投递可追踪** | 每次投递尝试记录 delivery_logs，通知状态全程可查 |
| **投递延迟** | PG LISTEN/NOTIFY 实时触发（~500ms）+ 轮询兜底（5s），双重保障 |
| **多实例并发安全** | PG 行级 CAS 抢占，同一条通知只被一个实例投递 |
| **供应商运行时管理** | 供应商配置 CRUD API，增删改无需重启服务 |

### 2. 明确选择不解决的问题及原因

| 不解决的问题 | 原因 |
|---|---|
| **外部 API 返回值的业务语义处理** | 需求明确说"业务系统不需要关心外部 API 的返回值"。本系统只关心投递是否成功（2xx），不解析响应体中的业务含义。如果未来需要，应由业务系统自己订阅投递结果回调，而非本系统代为处理 |
| **Exactly-Once 投递语义** | HTTP 调用天生不具备两阶段提交能力，exactly-once 在跨网络场景下不可实现。At-Least-Once + 消费方幂等是业内唯一务实的选择 |
| **供应商认证密钥的安全存储** | 供应商密钥（`vendor_configs.secret`）当前以明文存库。加密存储、密钥轮换、Vault 集成等属于基础设施安全层面的能力，不应耦合在通知服务中，应由企业的密钥管理基础设施统一解决 |
| **业务事件的触发逻辑** | 本系统是被动接收者（业务系统调 API 提交通知），不负责"何时该发通知"的判断。比如"用户注册后该通知哪个广告系统"属于业务规则，应由业务系统决定 |
| **请求限流 / 流量控制** | 当前未实现按业务系统维度的限流。限流策略与业务强相关（不同业务方配额不同），适合在 API 网关层统一管控，而非通知服务内部实现 |
| **消息优先级** | 所有通知按 `next_retry_at` 时间顺序投递，不支持业务方指定优先级。优先级调度会显著增加系统复杂度（优先级队列饥饿问题），当前业务场景无此强需求 |
| **投递结果的业务回调** | 本系统不主动回调业务系统告知投递结果。业务系统可通过 `GET /api/v1/notifications/{id}` 主动查询。主动回调意味着需要管理回调地址、重试、幂等，本质上是另一个通知投递问题，会让系统职责模糊 |
| **消息内容的校验与过滤** | 本系统不对 `payload` 内容做业务合法性校验，只负责透传到模板渲染。内容合规审查（如敏感词过滤）属于业务域职责 |
| **集群级别的分布式锁** | 当前通过 PG 行级 CAS 保证同一通知只被一个实例投递，未引入 Redis 分布式锁。PG CAS 在当前规模下足够，引入 Redis 会增加运维复杂度和故障点。如果未来实例数超过 10+，再考虑引入 |
| **投递顺序保证** | 同一业务方的多条通知不保证严格有序。保证有序需要串行化投递，会严重影响吞吐量。消费方应通过 `biz_id` + 时间戳自行处理乱序 |

## 四、可靠性与失败处理

### 1. 通知投递语义：At-Least-Once（至少一次）

#### 为什么选择 At-Least-Once

| 候选语义 | 可行性 | 分析 |
|---|---|---|
| **Exactly-Once** | 不可行 | HTTP 调用是单次网络请求，没有两阶段提交能力。请求发出后，无论收到响应还是超时，都无法确定对方是否已处理。强行实现 only-once 需要对方支持事务/幂等令牌，超出了本系统的控制范围 |
| **At-Most-Once** | 不满足需求 | 至多一次意味着失败不重试，与"可靠投递"的核心需求矛盾 |
| **At-Least-Once** | **唯一务实选择** | 通过持久化 + 重试保证消息不丢，代价是消费方可能收到重复通知，由消费方幂等兜底 |

#### At-Least-Once 在本系统中的保障机制

```
保障层           实现方式
─────────────────────────────────────────────────────
1. 消息不丢      写入 notifications 表后才返回，持久化优先
2. 投递必重试    失败后按指数退避重试，最多 6 次
3. 重试不遗漏    PG LISTEN/NOTIFY 实时触发 + 轮询兜底双重保障
4. 终态不丢失    死信持久化保存，支持手动重投递
5. 幂等由消费方  通知中携带 biz_id + notification_id，消费方据此去重
```

#### 什么场景下会产生重复投递

| 场景 | 原因 | 处理 |
|---|---|---|
| 外部系统返回 2xx，但本系统未收到响应（网络中断） | 本系统判定为超时，触发重试，外部系统实际已收到 | 消费方用 `notification_id` 去重 |
| 外部系统处理慢，本系统超时后重试 | 同一通知被外部系统收到两次 | 消费方用 `notification_id` 去重 |
| 多实例部署下 CAS 竞争边界情况 | 极端情况下同一条通知被两个实例尝试投递 | CAS 已最大程度避免，消费方仍需幂等 |

### 2. 外部系统失败或长期不可用的处理策略

#### 按失败类型分级处理

```
投递失败
  │
  ├─ 4xx 客户端错误（404/401/403...）
  │    │ 原因：URL 错误、鉴权失败、请求格式错误
  │    │ 判定：重试无意义，问题在本系统配置或对方接口变更
  │    └─→ 直接进入死信，通知运维修复配置
  │
  ├─ 5xx 服务端错误 / 超时 / 连接拒绝
  │    │ 原因：对方服务故障、过载、网络抖动
  │    │ 判定：可能是临时故障，有恢复可能
  │    └─→ 进入重试流程（指数退避 + Jitter）
  │
  └─ 熔断器 OPEN
       │ 原因：连续失败次数达阈值（10 次中 60% 失败）
       │ 判定：对方极可能已持续故障
       └─→ 跳过 HTTP 调用，直接计算退避等待
```

#### 短期故障：指数退避 + Jitter 重试

```
投递失败 → retry_count + 1
  │
  ├─ retry_count < max_retries (6)?
  │     ├─ 是 → 计算退避时间：base * 2^attempt + random_jitter
  │     │       status = RETRYING
  │     │       next_retry_at = now + 退避时间
  │     │       等待调度引擎在 next_retry_at 时再次投递
  │     │
  │     └─ 否 → 进入死信（见下方）
  │
  重试时间线（base=10s）：
  1st:  ~10s
  2nd:  ~30s
  3rd:  ~1min
  4th:  ~5min
  5th:  ~15min
  6th:  ~1h  ← 仍失败则进入死信
```

**Jitter 的作用**：当多个通知同时失败时，如果退避时间完全相同，它们会在同一时刻重试，产生"惊群效应"打向已过载的外部系统。Jitter 为每个通知的退避时间增加随机偏移，将重试请求在时间上打散。

#### 持续故障：熔断器保护

```
供应商 ad-system 连续失败：
                                                                   
  请求1 失败 ─┐                                                    
  请求2 失败  │ 10 次中 6 次失败                                    
  请求3 成功  │ → 失败率 60% 达阈值                                 
  请求4 失败 ─┘                                                    
       │                                                          
       ▼                                                          
  ┌─────────┐                                                     
  │  OPEN   │  持续 60s，所有投递请求直接跳过 HTTP 调用
  │  熔断中  │  按重试处理（计算退避等待），不再浪费连接和线程
  └────┬────┘                                                     
       │ 60s 冷却期结束                                             
       ▼                                                          
  ┌──────────┐                                                    
  │ HALF-OPEN│  放行 1 个探测请求                                   
  │  半开状态  │                                                    
  └────┬─────┘                                                    
       │                                                          
   ┌───┴───┐                                                      
   │       │                                                      
 成功    失败                                                     
   │       │                                                      
   ▼       ▼                                                      
 CLOSED   OPEN   ← 回到熔断，再等 60s
 (恢复正常)
```

**关键保护效果**：
- 外部系统故障期间，本系统**不会产生大量超时连接**占满连接池
- Worker 线程**不会被阻塞在等待超时上**，可以处理其他供应商的投递
- 退避重试的通知会在熔断恢复后自动投递，**不丢消息**

#### 长期不可用：死信 + 运维介入

```
6 次重试仍失败 或 4xx 客户端错误
  │
  ▼
status = DEAD_LETTER（终态）
  │
  ├─ 持久化保存：通知数据不丢弃，随时可查
  ├─ 记录 last_error：最后一次失败的错误信息
  ├─ 触发告警：日志 + Prometheus 指标 + 告警规则
  │
  └─ 运维修复后手动重投递：
     POST /api/v1/notifications/{id}/retry
     → status 重置为 PENDING
     → retry_count 重置为 0
     → 重新进入正常投递流程
```

#### 完整策略全景

```
外部系统响应
  │
  ├─ 2xx ────────────────────→ SUCCESS ✅
  │
  ├─ 4xx ────────────────────→ DEAD_LETTER ❌（不重试，运维修配置）
  │
  ├─ 5xx / 超时 / 连接拒绝 ──→ RETRYING 🔄
  │     │                         │
  │     │   指数退避 + Jitter     │
  │     │   10s → 30s → 1min → 5min → 15min → 1h
  │     │                         │
  │     │   重试 < 6 次 ─────────┘（继续重试）
  │     │   重试 ≥ 6 次 ─────────→ DEAD_LETTER ❌
  │     │
  │     └─ 熔断器 OPEN ──→ 跳过 HTTP，直接走退避 🛡️
  │
  └─ 熔断期间所有请求 ──────→ 跳过 HTTP，直接走退避 🛡️
                              冷却 60s 后半开探测
                              探测成功 → 恢复正常投递
                              探测失败 → 继续熔断
```

## 五、系统的取舍与演进

### 1. 流量或复杂度显著增长时的演进路径

#### 演进阶段总览

```
V1（当前）          V2（10x 流量）           V3（100x 流量）
─────────────      ──────────────────      ──────────────────
PG Outbox +        引入 MQ 解耦             MQ + 分片 + 多集群
轮询+LISTEN        缓解 PG 扫描压力          应对海量通知
```

#### V1 → V2：PG Outbox 扛不住时

**触发条件**：通知量级从千/天增长到万~十万/天，PG 的轮询扫描和 LISTEN/NOTIFY 成为瓶颈。

| 当前方案 | 演进方案 | 原因 |
|---|---|---|
| PG LISTEN/NOTIFY 触发投递 | 引入 Kafka/RabbitMQ | LISTEN/NOTIFY 是单通道广播，无法做消费组、无法做背压、无法堆积。MQ 天然支持消费组、分区、持久化堆积 |
| 轮询扫描 notifications 表 | MQ 消费者直接消费 | 轮询在大数据量下对 PG 造成压力，MQ 消费者模式是推式，无需轮询 |
| PG 行级 CAS 抢占 | MQ 分区消费天然互斥 | 每个 partition 只被一个消费者消费，不需要 CAS 竞争 |
| notifications 表既是存储又是队列 | 存储和队列职责分离 | notifications 表退回纯存储角色，不再承担调度职责 |

**迁移方式**：平滑升级，不是替换。NotificationService.submit() 写完 PG 后，同时发一条消息到 MQ，调度引擎改为从 MQ 消费。轮询保留作为兜底。MQ 稳定后可关闭轮询。

```
V1:  业务系统 → API → PG(notifications) → 轮询/LISTEN → 投递
V2:  业务系统 → API → PG(notifications) → MQ → 消费者 → 投递
                                      ↘ 轮询(兜底，可关闭)
```

#### V2 → V3：单集群扛不住时

**触发条件**：通知量级达到百万/天，单 PG 实例写入瓶颈，单 MQ 集群分区数上限。

| 维度 | 演进方案 |
|---|---|
| **数据库** | PG 分库分表（按 vendor_code 分片），或迁移到 TiDB/CockroachDB 等 NewSQL |
| **MQ** | Kafka 按 vendor_code 分区，不同供应商的通知投递到不同分区，天然隔离 |
| **投递服务** | 按供应商分组部署独立消费组，广告通知投递组和 CRM 通知投递组物理隔离 |
| **供应商配置** | 从 PG 迁移到配置中心（Nacos/Apollo），支持热更新和多集群同步 |
| **限流** | 引入 Redis 令牌桶限流，按 vendor + 业务系统维度精细控制 |
| **观测** | 从 Prometheus + Grafana 升级到 OpenTelemetry + 分布式追踪 |

#### 演进中始终不变的部分

| 不变 | 原因 |
|---|---|
| At-Least-Once 投递语义 | 无论底层是 PG 还是 MQ，HTTP 投递的语义约束不变 |
| 投递状态机（PENDING→SENDING→SUCCESS/RETRYING/DEAD_LETTER） | 状态机是业务逻辑，与存储和调度无关 |
| 模板化供应商适配 | 适配逻辑与流量规模无关 |
| 熔断器保护 | 外部系统故障保护逻辑与架构无关 |
| 死信 + 手动重投递 | 终态兜底机制与底层调度无关 |

### 2. 被判定为“过度设计”而未采纳的方案

| 被否决的设计 | 看似的好处 | 被否决的原因 |
|---|---|---|
| **优先级队列（业务方指定通知优先级）** | 紧急通知优先投递 | 引入优先级后必须处理饥饿问题（低优先级永远排不到），需要复杂的调度策略。当前业务场景所有通知同等重要，无优先级需求。如果未来有，可在 MQ 层按分区实现，而非在应用层实现 |
| **投递结果主动回调业务系统** | 业务方无需轮询查询结果 | 这等于在本系统内又实现一个通知投递系统（管理回调地址、回调重试、回调幂等），职责模糊。业务方主动查询（`GET /notifications/{id}`）在 V1 阶段足够。如果未来需要，可引入 SSE/WebSocket 推送状态变更，而非 HTTP 回调 |
| **消息内容校验与敏感词过滤** | 防止不合规内容发送到外部系统 | 内容合规是业务域问题，不同业务方合规标准不同。放入通知服务会把业务逻辑耦合进来，导致每个业务方的合规规则变更都需要改通知服务 |
| **分布式锁（Redis）替代 PG CAS** | 更高并发下的抢占性能 | 当前 PG 行级 CAS 完全够用。引入 Redis 意味着新增一个中间件、新增一个故障点、新增运维成本。在实例数 < 10 的规模下，PG CAS 的竞争损耗可忽略不计 |
| **消息顺序保证** | 同一业务方的通知按顺序投递 | 保证有序需要串行化投递（同一 vendor 只能有一个 Worker 处理），严重限制吞吐量。通知场景对顺序不敏感（"注册通知"和"付款通知"谁先到都行），消费方通过时间戳 + biz_id 自行处理乱序即可 |
| **供应商密钥加密存储（Vault 集成）** | 安全性更高 | 这是基础设施安全层面的能力，不应耦合在通知服务中。当前密钥存 PG、网络层加密（TLS + VPC 隔离）在 V1 阶段足够。加密存储应交给企业的密钥管理基础设施统一解决 |
| **GraphQL 或复杂查询 API** | 业务方灵活查询通知 | V1 阶段查询需求简单（按 ID 查、按状态列表），REST API 足够。GraphQL 增加实现复杂度且带来 N+1 查询风险，收益不覆盖成本 |
| **事件溯源（Event Sourcing）** | 完整的事件历史可回放 | delivery_logs 已记录每次投递尝试，足以满足审计和排查需求。完整的事件溯源需要事件存储 + 快照 + 投影，复杂度远超当前需求 |

#### 判断“是否过度”的原则

```
                    收益高
                      │
           采纳        │      采纳并简化
                      │
 ─────────────────────┼───────────────────── 
        复杂度低       │       复杂度高
                      │
           否决        │      延迟到 V2+
                      │
                    收益低
```

- **复杂度低 + 收益高** → 直接采纳（如模板化配置、指数退避、熔断器）
- **复杂度高 + 收益高** → 简化后采纳（如 LISTEN/NOTIFY 用轮询替代真异步、CAS 替代分布式锁）
- **复杂度低 + 收益低** → 直接否决（如 GraphQL、优先级队列）
- **复杂度高 + 收益低** → 延迟到规模增长时再考虑（如 MQ 替代 PG 调度、密钥加密存储、分布式锁）

> V1 的设计哲学是**"够用即止"**——用最简单的方案满足核心需求（可靠投递），把复杂度留给未来有明确需求时再引入。每一个被否决的设计都不是"永远不做"，而是"当前阶段收益不足以覆盖复杂度"。
