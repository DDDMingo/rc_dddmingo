# 模块职责说明

## 1. API 接收层

| 子模块 | 职责 |
|---|---|
| **路由注册** | 定义 RESTful API 端点（提交通知、查询状态、管理供应商配置等） |
| **参数校验** | 请求体格式校验、必填字段检查、供应商编码有效性验证 |
| **幂等处理** | 基于 `biz_id` 唯一约束，防止业务方重复提交产生重复通知 |
| **限流** | 按业务系统/供应商维度限流，防止突发流量压垮下游 |
| **响应封装** | 统一响应格式，成功返回 202 + notification_id，失败返回结构化错误码 |

## 2. 持久化层

| 子模块 | 职责 |
|---|---|
| **notifications 表** | 核心 Outbox 表，存储通知消息、投递状态、重试计数、下次重试时间 |
| **delivery_logs 表** | 每次投递尝试的详细记录（状态码、响应体、耗时、错误信息），用于排查和审计 |
| **vendor_configs 表** | 供应商配置（URL、Header/Body 模板、签名密钥、超时时间、最大重试次数） |
| **数据库迁移** | Schema 版本管理，使用 golang-migrate 管理 DDL 变更 |
| **LISTEN/NOTIFY** | 写入 notifications 后触发 PG NOTIFY，实时通知调度引擎有新消息待投递 |

## 3. 调度引擎

| 子模块 | 职责 |
|---|---|
| **PG LISTENER** | 监听 PG NOTIFY 通道，收到通知后立即将消息推入优先级队列，实现 ms 级投递延迟 |
| **轮询扫描器** | 定时（~5s）扫描 notifications 表中 `next_retry_at <= now()` 的记录，兜底补偿 NOTIFY 丢失或实例启动时的遗漏 |
| **优先级队列** | 小顶堆实现，按 `next_retry_at` 排序，确保最早该投递的消息优先出队 |
| **CAS 抢占** | 通过 `UPDATE ... WHERE status='pending' RETURNING *` 实现行级锁，保证多实例下同一通知只被一个 Worker 处理 |
| **Worker 池** | 可配置并发数的 goroutine 池，从优先级队列消费任务，调用投递层执行实际投递 |

## 4. 投递层

| 子模块 | 职责 |
|---|---|
| **模板渲染器** | 从 vendor_configs 加载模板，将业务 payload 中的字段渲染到 Header 和 Body 模板中，生成最终 HTTP 请求 |
| **HMAC 签名** | 对渲染后的请求体计算 HMAC-SHA256 签名，写入 Header，供消费方验证请求来源和完整性 |
| **熔断器组** | 每个供应商一个独立 gobreaker 实例，按 vendor_code 路由。Closed 正常投递，Open 拒绝请求直接走退避，HalfOpen 放行探测请求 |
| **HTTP 客户端** | resty 发起实际 HTTP 请求，配置超时、连接池、重试次数为 0（重试由调度引擎控制，不由客户端控制） |
| **响应判定** | 根据 HTTP 状态码判定投递结果：2xx 成功，4xx 客户端错误进死信，5xx/超时进入重试 |

## 5. 异常处理

| 子模块 | 职责 |
|---|---|
| **指数退避计算** | 根据重试次数计算下次重试时间：`base * 2^n + random_jitter`，避免惊群效应 |
| **退避时间写入** | 更新 notifications 表的 `next_retry_at` 和 `retry_count`，使消息重新进入调度队列 |
| **死信处理** | 超过最大重试次数或收到 4xx 的通知，标记为 `dead_letter` 终态，不再重试 |
| **死信告警** | 产生死信时触发告警（日志 + 指标），通知运维人员介入 |
| **手动重投递** | 提供管理 API，允许运维修复问题后手动将死信消息重新投递（状态重置为 pending） |

## 6. 观测层

| 子模块 | 职责 |
|---|---|
| **结构化日志** | zap 输出结构化日志，每条日志携带 notification_id、vendor_code、retry_count 等字段，支持链路追踪 |
| **投递指标** | Prometheus 暴露核心指标：投递总数/成功数/失败数（按 vendor/status 维度）、投递延迟分布、重试次数分布、熔断器状态 |
| **队列深度监控** | 监控优先级队列中待投递消息数量，队列积压时告警 |
| **Grafana 大盘** | 可视化展示投递成功率、P95/P99 延迟、重试趋势、各供应商健康状态 |
| **告警规则** | 死信产生速率异常、熔断触发、队列深度超阈值、投递成功率低于阈值 |

## 模块间调用关系

```
API 接收层
  │ 写入
  ▼
持久化层 ──NOTIFY──▶ 调度引擎
                        │ CAS 抢占 + 出队
                        ▼
                     投递层
                   ╱       ╲
              成功             失败
               │               │
               ▼               ▼
          持久化层        异常处理
          (更新状态)    (退避/死信)
                             │
                             ▼
                         持久化层
                       (更新状态)

全链路 ◀─────── 观测层（日志 + 指标 + 告警）────────▶ 贯穿所有模块
```
